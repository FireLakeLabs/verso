Prototype feasibility and gap analysis

Feasibility is strong if this prototype is treated as a binding design reference for the screens it actually covers, not as a complete Solid v1 product spec. The architecture and current frontend scaffold are compatible with the design direction, but the issue backlog needs cleanup before implementation starts in earnest.

Feasibility
High for covered screens

The prototype was handed off for the same core stack Verso already chose: Vite + React + shadcn/ui + TypeScript on the frontend, local ASP.NET Core + SQLite on the backend.

Backlog fit
Partial, not complete

Several issues align well, but multiple issue bodies are either broader than the prototype, narrower than the signed-off UI, or missing a design entirely.

Main risk
Ownership gaps

Item Detail, UI preference persistence, export-job UX, and several Solid v1 reports do not have a clean one-to-one issue-to-design mapping yet.

Recommendation: accept this prototype as the authoritative visual target for the already-modeled surfaces, then update the issue backlog before implementation to avoid teams inventing missing flows or building against contradictory scope.
Executive read

What is feasible right now

  • The current frontend scaffold can absorb the Firelake Labs visual system without an architecture change. The existing shadcn/Tailwind CSS-variable setup is a workable base for remapping typography, colors, spacing, and radii.
  • The prototype matches the repo's big architectural choices: local two-server app shape, backend-owned refresh and health evaluation, frontend-owned report transforms, cover caching, and settings as a local-only surface.
  • The prototype is not a complete Solid v1 screen inventory. Missing or underspecified areas include Listening Cadence, Genre Treemap, Narrator Affinity, Cost-per-Hour, Smart Shelves, export job status, dedicated tag-management UI, command palette behavior, and robust empty/error states.
  • The biggest issue backlog problem is not visual feasibility. It is ownership clarity. Several shipped prototype surfaces do not have a clean issue owner, while some issues bundle one designed screen with one undesigned screen.

Prototype gaps

What the current prototype still does not cover

Missing screens or control flows

  • Listening Cadence Heatmap
  • Genre Treemap
  • Narrator Affinity report
  • Cost-per-Hour report
  • Smart Shelf list, rule builder, and preview flows
  • Archive export job status, progress, completion, and errors
  • Dedicated tag management or tag add/remove dialogs

Cross-cutting gaps

  • In-product controls for nav style and overview preference; today those live only in the design-tool tweak panel.
  • Command palette behavior, despite a visible ⌘K affordance in the shell.
  • Systematic empty, zero-data, and error-state coverage for refresh, export, health, and report screens.
  • Responsive or narrow-viewport treatment. The handoff is desktop-first only.
The Firelake Labs design skill is a good fit for generating the missing report screens, Smart Shelf builder, export-job surface, and a small preference settings treatment that replaces the tweak panel in production.
Direct incompatibilities

Design, issue, and prototype mismatches that must be resolved

  • Bulk tagging scope: #6 says bulk tagging can target selected or filtered items; the prototype only shows actions for selected rows.
  • Health dispositions: #7 defines stable dispositions, but the prototype adds Snooze without a corresponding model or lifecycle.
  • Cover-wall sorts: #15 and the prototype disagree about which sorts matter.
  • Combined report issues: #10, #11, and #12 each bundle one designed report with one undesigned report.
  • Overview and Item Detail ownership: the handoff treats them as part of the Solid v1 baseline, but no current issue body names their full UI scope clearly enough.
  • Preference controls: nav style and overview variant are required behaviors in the handoff, but they are not currently represented as a product-owned settings surface.
  • Export job UX: #16 requires a local job with status and typed errors, but the prototype does not show that UI yet.
Screen review

Per-section analysis with prototype evidence

Navigation shell

Feasible, but currently unowned

  • Both top-nav and sidebar patterns are visually complete enough to implement.
  • No issue clearly owns the production control for switching between them or persisting the preference.
  • This belongs either in #8 or in a small follow-up issue from the closed scaffold work.
  • The current frontend scaffold is compatible with this shell, but the Firelake token remap should happen before shipping a large batch of stock shadcn components.
Library overview

Strong product fit, but the issue body is too narrow

  • The design clearly matches the intended first-screen experience from the ADRs and is practical to build.
  • #5 should explicitly own both overview variants, report entry points, refresh summary, and the links into Library Table, Health, and Refresh Status.
  • The dense variant contains mini cadence and narrator previews. Those support discovery, but they do not replace the missing full designs for Listening Cadence and Narrator Affinity.
  • The user preference for calm vs dense needs a real product control, not just the prototype tweak panel.
Library table

Mostly aligned, with one direct issue mismatch

  • The screen is ready to drive implementation of search, filtering, sorting, view toggles, pagination, dropped visibility, and row selection.
  • The mismatch is in #6: the issue allows bulk actions for selected or filtered items, but the designed UI only supports selected items.
  • The visible Manage tags action has no designed destination. Decide whether tag CRUD remains inline or whether the app needs a dedicated tag-management surface.
  • Both rows and cards look shippable. Keep both. The product already implies a user-facing mode choice.
Item detail

High-value screen with no clean backlog owner

  • This screen is detailed enough to implement and is important to the overall product quality.
  • No current issue owns it cleanly. The best fix is either to expand #5 or add a dedicated issue for Item Detail and snapshot-history presentation.
  • #3 should call out the DTO fields this screen depends on: category ladder, keywords, plans, returnability, series metadata, summary, price, dates, and cover-asset references.
  • #6 also touches this screen because tags and dropped marking are visible here, not only in the Library Table.
Cover wall

Ready to implement after one product decision

  • This is a strong fit for #15 backed by #9.
  • The unresolved question is sorting: the signed-off design favors Recent, Runtime, Palette, and Random, while the issue text allows release date and title instead.
  • The prototype fallback covers work as graceful placeholders only. Production still needs local cached cover assets as the primary path.
Author concentration

Designed well, but bundled with an undesigned sibling report

  • This screen is complete enough to implement the author half of #12.
  • The same issue also owns Narrator Affinity, which is not designed here beyond a small dense-overview preview.
  • Split the issue or add a design task for the missing narrator screen before implementation starts.
Runtime distribution

Good to build now, but paired with a missing cadence report

  • This screen provides enough detail for the Runtime Distribution half of #10.
  • The Listening Cadence Heatmap still needs a dedicated design. The dense overview sparkline is only a teaser.
  • If implementation begins without splitting the issue, the team will either invent the cadence screen or block on design mid-stream.
Subject keywords

Strong fit for keywords, no coverage for treemap

  • This is a complete design for the Subject Keyword side of #11.
  • Genre Treemap still has no signed-off visual treatment in the bundle.
  • Either split the issue or explicitly queue a design step before the treemap work starts.
Library health

Very close, but the disposition model is not resolved

  • This is a good visual fit for #7 and the backend-authoritative findings model.
  • The main design/issue mismatch is the Snooze action. If snooze is real, the issue needs a time-based disposition model. If it is not real, the button should be removed from the design.
  • The issue also asks for active and dismissed findings to be clearly presented. The prototype mostly emphasizes active findings and an undo strip, but not a stable dismissed view.
Refresh status

One of the strongest alignments in the bundle

  • This screen fits #5 extremely well and reflects the ADR-backed refresh pipeline cleanly.
  • The issue should call out the dedicated refresh-status screen explicitly if it stays separate from the Overview and Library Table acceptance text.
  • The Schedule… entry and the Settings refresh panel surface the choice, but the operational meaning of Daily at idle still needs refinement in the backlog.
  • This pattern is also the obvious template for the still-undesigned export-job surface required by #16.
Settings

Mostly aligned, with one missing product-surface concept

  • The major settings areas in #8 are represented well: auth, refresh, cost basis, local data, and archive export defaults.
  • What is still missing is the real production home for shell and display preferences. Those exist only in the prototype tweak panel right now.
  • The refresh panel is a good place to clarify Daily at idle, retention policy, and the read-only snapshot field set.
  • Archive export options are visually present, but the export-job lifecycle and result handling required by #16 are not.
